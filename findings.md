# Test Orchestrator - Findings

## Research Insights
- **Target Users:** QA Engineers (primary), QA Leads/Managers (secondary), Developers (tertiary).
- **Scale:** Starts as Internal Tool (MVP: 10-50 concurrent users), built to scale to SaaS (500-2000 users, multi-project).
- **Differentiator:** Seamless flow from Jira Story → Test Plan → Test Cases → Automation Code.
- **Workflow:** AI outputs MUST pass through a mandatory "Review Layer" before final save.

## Technical Decisions
- **Repo Strategy:** Monorepo (Turborepo or Nx) with structured `/apps` and `/packages`.
- **Frontend:** Next.js (App Router, Server/Client components) + React Query + Tailwind CSS.
- **Backend:** NestJS (Modular Architecture: Auth, Jira, Test Plan, Test Case, Code Gen, AI).
- **Database:** PostgreSQL with Prisma ORM. Fully normalized schema (NO JSON blobs for core entities like test steps).
- **AI Provider:** OpenAI (GPT-4o) abstracted via interfaces. Prompts must be centralized.
- **Automation Framework:** Playwright (Primary) with POM. Must generate structured files (`/tests`, `/pages`, `/fixtures`) and prioritize `data-testid` selectors.
- **Security:** JWT auth, bcrypt password hashing, encrypted Jira tokens, AI API rate limiting.
- **Performance:** Async processing for AI, UI loading states, short-term Jira caching.

## Constraints
- **Jira Integration:** Cloud only for now. Authentication via per-user API Token.
- **MVP Boundaries (Avoid Overengineering):** No OAuth, no PDF export, no CI/CD triggers, no Execution tracking in MVP.

## Assumptions
- **Jira Data:** Fetching Summary, Description, Acceptance Criteria, Issue Type, Priority, and Labels. Multi-project support is mandatory.
- **Test Plans:** Hybrid format (IEEE 829 + Industry Template) including Objective, Scope, Strategy, Test Types, Environment, Data Strategy, Risks, Entry/Exit Criteria, Roles.
- **Test Cases:** Must include ID, Title, Description, Preconditions, Steps, Expected Results, Priority, Type (Pos/Neg/Edge), Tags, Linked Jira ID. Mix of Positive/Negative/Edge cases generated by default.
- **Dashboard:** Filters by Jira ID, Priority, Test Type, Module, Status, Date. Excel Export. Full inline editing.

## Risks
- **AI Hallucination:** Generating invalid or irrelevant test cases/code.
- **Jira API Rate Limits:** Hitting limits when syncing large projects.
- **Automation Stability:** Poor selector generation leading to flaky automation code.
- **Latency:** High response times during large test case/plan generation tasks.

## Phase 3E — Jira Integration: Technical Debt & Design Decisions

### 🔐 Credential Storage (MVP Decision)
- **Current:** Jira `apiToken` is stored as **plaintext** in the `JiraConnection` table.
- **Why Acceptable for MVP:** Internal tool with controlled user base.
- **Future (MANDATORY before SaaS):** Encrypt using **AES-256-GCM** (field-level) or delegate to a **KMS** (AWS KMS / GCP Cloud KMS).
  - Store only the encrypted ciphertext + IV in the DB.
  - Decrypt at runtime inside `JiraClient` before making API calls.
  - Key rotation must be supported without data loss.

### 🔁 Duplicate Story Prevention
- On `POST /api/jira/stories/:projectId`, stories are upserted by `jiraId`.
- Uses Prisma `upsert` with `where: { jiraId }` to avoid duplicate rows.
- **Requires** a `@@unique([jiraId])` constraint on the `UserStory` model (already present via `jiraId` field uniqueness).

### ⚡ Jira API Rate Limiting
- Jira Cloud REST API v3 enforces rate limits per OAuth/token.
- Future: Add short-lived cache (Redis or in-memory TTL map) on `GET /jira/projects` responses.
- Retry logic with exponential backoff should be added before production.

### 🌐 Scope: Jira Cloud Only (MVP)
- Authentication via per-user **API Token** (Basic Auth: `email:token` base64).
- No Jira Data Center / Server support in MVP.
- No OAuth 2.0 flow in MVP (planned for Phase 5+).
